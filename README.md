## Теоретические вопросы

### 1. Зачем выносить виджеты в отдельные файлы? Что изменится, если держать всё в `main.dart`?

**Зачем выносить:**

- **Читаемость и поддерживаемость:** Код становится модульным. Легче найти ошибку или изменить логику конкретного элемента интерфейса, не пролистывая сотни строк в одном файле.
- **Переиспользование (Reusability):** Виджет, вынесенный в отдельный файл (например, `GradientContainer`), можно легко импортировать и использовать в других экранах приложения.
- **Командная разработка:** Разные разработчики могут работать с разными файлами одновременно без конфликтов слияния (merge conflicts) в Git.

**Если держать всё в `main.dart`:**

- Файл быстро разрастется, станет нечитаемым («спагетти-код»).
- Усложнится навигация по коду.
- Нарушится принцип единственной ответственности (Single Responsibility Principle).
- Для небольших учебных проектов это допустимо, но в продакшене считается плохой практикой.

### 2. Что такое `BuildContext`? Почему метод `build()` принимает его как параметр?

**Что это:**
`BuildContext` — это абстракция, которая указывает на местоположение виджета в дереве виджетов (Widget Tree). Каждый виджет имеет свой собственный контекст.

**Зачем он в `build()`:**

- **Доступ к данным:** Через контекст виджет может получать данные от родительских виджетов (например, тему оформления `Theme.of(context)` или медиа-запросы `MediaQuery.of(context)`).
- **Навигация:** Позволяет открывать новые экраны (`Navigator.push(context, ...)`).
- **Поиск предков:** Позволяет обращаться к родительским виджетам определенного типа (например, `ScaffoldMessenger.of(context)` для показа Snackbar).

Без `BuildContext` виджет был бы изолирован и не знал бы, где он находится в приложении и какие ресурсы ему доступны.

### 3. Чем `StatelessWidget` отличается от `StatefulWidget`? Приведите пример, когда нужен каждый из них.

**StatelessWidget (Статический виджет):**

- Не имеет изменяемого внутреннего состояния.
- Перерисовывается только если родительский виджет передаст ему новые параметры (arguments).
- **Пример использования:** Заголовки, иконки, статические тексты, кнопки, которые не меняют свой вид при нажатии (только вызывают действие), контейнеры с фиксированным дизайном (как наш `GradientContainer`).

**StatefulWidget (Динамический виджет):**

- Имеет состояние (`State`), которое может меняться в течение жизни виджета.
- При изменении состояния (вызов `setState()`) виджет перестраивается, обновляя UI.
- **Пример использования:** Поля ввода текста, чекбоксы, анимации, счетчики, и в нашем случае — **кубик**, который меняет картинку при нажатии на кнопку.

### 4. Почему `Random()` создаётся на уровне файла/класса, а не внутри `rollDice()`?

_(Примечание: В предоставленном коде выше логика рандома была упрощена, но вопрос касается общей практики)_

**Причина:**

- **Производительность:** Создание объекта `Random()` — это операция выделения памяти. Если создавать его каждый раз при нажатии кнопки (внутри метода), это будет нагружать сборщик мусора (Garbage Collector), особенно при частых вызовах.
- **Качество случайности:** Объект `Random` инициализируется сидом (seed), часто на основе текущего времени. Если создавать новый экземпляр `Random()` очень быстро (в цикле или при быстрых кликах), они могут получить одинаковый сид и выдать одинаковые последовательности чисел.
- **Best Practice:** Рекомендуется создавать один экземпляр `final Random _random = Random();` на уровне класса состояния и переиспользовать его метод `.nextInt()`.
